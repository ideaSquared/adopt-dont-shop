# Major-version Upgrade Plans

This directory holds **planning documents** for the round of major-framework
upgrades flagged by the Production Readiness Audit (Linear: ADS-531, ADS-532).
Node 22, Express 5 and React 19 have all shipped; Sequelize is on 7-alpha. The
individual planning docs below retain the original scoping, risk inventory and
migration notes for posterity — re-read the **Status** banner at the top of
each one before treating it as live guidance.

## Sequencing recommendation

These upgrades are **independent in principle** but share infrastructure (CI,
Docker images, lockfile churn). The recommended ordering minimises rework:

1. **Node 22** first — it is the cheapest (image bump + CI matrix) and unblocks
   any future tooling that demands a newer V8/undici. Express 5, Sequelize 7
   and modern Vitest releases all benefit from being exercised on Node 22.
2. **Express 5** second — confined to `service.backend`, low blast radius
   compared with the frontend upgrade. Best done while the Node bump is still
   fresh in operators' minds.
3. **React 19** third — touches three apps and `lib.components`, which is the
   largest surface area. Best done after backend churn settles so the apps can
   regress against a stable API.
4. **Sequelize 7** last — Sequelize 7 is still in pre-release at audit time;
   we should wait for a stable tag and let the ecosystem (sequelize-typescript,
   adapters) catch up before committing.

Re-evaluate this ordering at the start of each upgrade — if the ecosystem
lands a forcing function (e.g. a CVE in Sequelize 6) we promote that upgrade.

## How to use these docs

- Each plan is structured the same way: current state → breaking changes →
  risk inventory → migration path → effort → test/rollback → prerequisites →
  sub-issues.
- Numbers (file counts, occurrences) were generated by `grep` against this
  worktree at audit time. Re-run the listed commands before starting work to
  pick up drift.
- Every plan ends with a list of **Linear sub-issue titles**. File those when
  the corresponding upgrade is scheduled into a sprint — do not file them up
  front because they will go stale.

## Out of scope for the audit PR

- Bumping any version in `package.json`.
- Touching migrations, models, Dockerfiles or CI workflow files.
- ADS-414 cleanup work (tracked separately).
